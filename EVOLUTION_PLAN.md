# 航空气象 Agent 进化方案

> 设计日期: 2026-04-14

## 文档目标

本文档只定义三件事：
1. 这个 Agent 的真实端到端任务是什么
2. 这个 Agent 应该用什么指标验证价值
3. 这个 Agent 的优化结果最后应该如何体现

---

## 标准端到端任务

标准例子：
- 角色：飞行员
- 问题：虹桥机场天气怎么样
- 期望过程：
  - 读取机场天气 API
  - 提取对飞行员有用的信息
  - 根据飞行员规则给出建议
- 期望输出：
  - 简洁自然语言建议
  - 不使用固定模板
  - 不输出与当前问题无关的大段说明

完整链路：

```
用户选择角色（飞行员）
→ 输入问题（虹桥机场天气怎么样）
→ Agent 判断需要获取实时机场天气
→ 调用天气 API
→ 读取并整理天气结果
→ 按飞行员关心点生成建议
→ 输出自然语言结论
→ 自动评测这次结果是否完成任务、是否抓住关键风险、是否模板化
→ 若失败则沉淀为 badcase
```

系统输出的目标不是生成固定模板报告，而是基于当前问题、当前角色、当前天气数据，给出最小但足够使用的建议。

## 赋能业务场景

四个角色，每个角色有一个核心场景：

| 角色 | 核心场景 | 他问什么 | 他要什么 |
|------|---------|---------|---------|
| 飞行员 | 进近决策 | "虹桥能降吗" | 结论：能不能降，用什么方式降 |
| 签派员 | 放行决策 | "ZBAA飞ZSPD能飞吗" | 结论：能不能飞，带多少备降油 |
| 预报员 | 天气分析 | "ZBAA未来趋势" | 结论：会变好还是变差，什么时间窗 |
| 地勤机务 | 作业决策 | "现在能出外场吗" | 结论：能还是不能，什么限制 |

不要为每个角色写一套模板。核心是：用户问什么，Agent 就基于当前数据回答什么。角色只是影响关注点，不是影响输出格式。

## 案例库构建逻辑

每条 case = 一个 METAR + 一个角色 + 一个问题 + 一条期望输出。

```
每条 case = {
  metar: "METAR ZSSS 140600Z 18012KT 0800 FG BKN002 ...",
  role: "pilot",
  query: "虹桥机场天气怎么样",
  expected_key_info: ["能见度800m", "有雾", "云底高200ft", "CAT II满足"],
  expected_flight_rules: "LIFR",
  scoring_criteria: {
    必须提到: ["能见度<1km", "云底高<300ft"],
    必须给出: ["进近建议"],
    禁止输出: ["固定模板栏目"]
  }
}
```

案例库的分类维度不是"VFR/MVFR/IFR"这种气象分类，而是：
- **按角色**：飞行员问什么、签派问什么、预报员问什么、地勤问什么
- **按问题类型**：当前天气、趋势判断、起降建议、作业风险
- **按复杂度**：单机场直问、多机场对比、边界天气、极端天气

## 打分规则设计

四个主指标，每个拆成可自动判定的子项：

### 任务完成率 — 脚本判定

case.query 是否被回答了？
→ 不是返回了数据就算完成，必须有结论。
→ 关键词/正则/LLM-judge 判断输出里有没有"回答"用户的问题。

### 关键信息命中率 — 逐条对比

每条 case 预标注期望信息：
```
case.expected_key_info = ["能见度800m", "ILS CAT II满足"]
→ 逐条检查输出里是否提到
→ 命中数 / 总数 = 命中率
```

### 输出可用率 — LLM-as-Judge，只评3个问题

1. 这个输出能不能直接给飞行员用？(是/否)
2. 有没有说了很多但没给建议？(是/否)
3. 是不是在套模板？(是/否)

### badcase 回归通过率 — 最硬的指标

历史失败 case 列表 → 本轮全部重跑 → 通过数 / 总数。

核心原则：**能脚本判的不靠 LLM，能靠 LLM 的不靠人，靠人的只抽检不全量。**

## 评测指标

### 主指标

1. **任务完成率**
定义：用户的问题是否被真正完成，而不是只返回中间结果。

2. **关键信息命中率**
定义：当前角色最关心的信息是否被说到。
以飞行员为例：能见度、云底高、风、起降相关风险，而不是泛泛复述天气。

3. **输出可用率**
定义：输出是否能直接拿给角色使用。
失败表现：模板腔太重、说了很多但没有建议、和角色无关、没回答用户的问题。

4. **badcase 回归通过率**
定义：历史失败样本修复后是否真的不再复发。

### 辅助指标

5. **幻觉率**
定义：编造数据、编造规则、编造建议。

6. **延迟 / 成本**
定义：保证优化不会把系统改得又慢又贵。

### 不作为主指标的项目

- METAR 字段解析准确率
- 静态规则判断准确率
- 单独的角色分类得分
- 上下文精度这类宽泛指标

这些内容只保留"脚本可验证、回归可拦截"即可，不进入首页 KPI。

## 输出约束

系统输出必须满足：
- 先回答用户当前问题
- 只保留与当前角色相关的信息
- 给出结论或建议
- 不生成固定模板化报告
- 不为了"显得专业"而堆砌长段结构

反例：
- 统一输出【风险分析】【建议措施】【角色职责】等固定栏目
- 不管用户问什么都输出整套格式
- 用长篇模板替代真实建议

系统不要求为不同角色输出不同模板，而要求：
- 命中该角色最关心的信息
- 给出该角色最需要的建议
- 输出尽量短、尽量直接、尽量少废话

---

## badcase 自动回流

当出现以下任一情况时，自动沉淀为 badcase：
- 任务未完成
- 没抓住关键风险
- 输出模板化严重
- 输出不可直接使用
- 出现明显幻觉

所有 badcase 自动进入回归集。
下一轮优化后，必须优先验证这些 badcase 是否被修复。

badcase 回归通过率是系统是否真正进化的核心指标之一。

## 评测集构建

```
eval/datasets/
├── standard_testset_v1.json     # 50条标准集 (git tag冻结，改需PR)
├── boundary_testset_v1.json     # 100条边界集 (季度更新)
├── adversarial_testset_v1.json  # 30条对抗集 (持续更新，开发不可见)
└── README.md                    # 数据字典+标注规范

eval/golden_answers/
├── standard_v1_answers.json     # 标准集人工标注答案
└── boundary_v1_answers.json     # 边界集人工标注答案

eval/results/
└── eval_2026-04-14_run01.json   # 每次评测结果存档
```

防过拟合措施：
1. 标准集 git tag 锁定，修改走 PR + 人工审核
2. 边界集从真实历史 METAR 分层抽样，非人工编造
3. 对抗集独立管理，开发团队不可见
4. hold-out 集 10 条永不参与调优，发版前 100% 人审

## 自动化评测流水线

```
开发者 git push
    ↓
pre-push hook触发 (~3分钟)
    ↓
┌─────────────────────────────────────────┐
│ L1 标准集 (50条，必须全部通过)            │
│ L2 边界集 (100条，允许5%失败)             │
│ L3 抽样20条 (幻觉+关键信息+可用性)        │
└─────────────────────────────────────────┘
    ↓
  L1全部通过 + L2失败<5% + L3通过 → ✅ 允许push
  否则 → ❌ 拒绝push + 输出失败详情
```

评测频率：

| 场景 | 跑哪些集 | 耗时 |
|------|---------|------|
| git push | L1+L2+L3抽样 | ~3分钟 |
| PR merge | 全部+对抗集 | ~5分钟 |
| 发版前 | 全部+hold-out+人工质检 | ~15分钟 |

## 人工质检环节

| 类别 | 触发条件 | 频率 |
|------|---------|------|
| 安全类 | 任何安全边界相关 case | 每次评测 |
| 幻觉类 | 幻觉标记 > 0.5 | 每周抽检 |
| 新类型 | 首次出现的天气组合 | 实时 |
| 争议 case | L1 标准集失败 | 每次评测 |
| Hold-out | 10 条未参与训练的 case | 发版前 |

## 温度按任务配置

| 任务场景 | 温度 | 原因 |
|----------|------|------|
| METAR 解析/数据抽取 | 0.0 | 需要精确，不能随机 |
| 风险评估决策 | 0.0 | GO/NO-GO 不能有随机性 |
| 角色识别 | 0.3 | 分类任务 |
| 自然语言报告生成 | 0.7 | 需要流畅 |
| 评测裁判 (LLM-as-Judge) | 0.0 | 裁判必须确定性 |

代码变更：`app/agent/graph.py` 新增 `get_llm_for_task()` 按任务类型选温度。

## PE 改写

**旧 PE (V2 Pipeline, 100行固定模板)：**
```
你是资深航线飞行员...
【核心职责】- 评估能见度...
【思维链路】步骤1...步骤2...步骤3...步骤4...
【输出格式】严格输出JSON：{"flight_decision": {"go_no_go": "..."}}
```
问题：写死流程和输出格式，LLM 只是填空机器。

**新 PE (V3 Agent, 约束边界)：**
```
你是一名资深航线飞行员（PIC），持有ATPL执照，累计飞行15000+小时。

## 你的关注范围
  - 能见度、云底高、风向风速对起降的影响
  - 进近方式选择（ILS CAT I/II/III, VOR, NDB）

## 你不需要涉及的内容
  - 签派放行决策（延误、取消、条件放行）
  - 地面作业建议

## 工作方式
- 你可以使用工具获取实时METAR数据
- 根据用户的具体问题，自主决定需要调用哪些工具
- 不要用模板套话，每次回答都要基于当前具体数据
```

改写逻辑：
1. 砍掉 CoT 步骤 → Agent 自己决定思考路径
2. 砍掉输出格式 → LLM 根据问题复杂度自适应
3. 加约束边界 → 只定义"你是谁"+"你关注什么"+"你别碰什么"
4. 加工具指引 → 告诉 LLM 可用工具，不规定顺序

## 支撑层（简写）

可观测性、记忆、监控的作用：
- 帮助回溯 Agent 过程
- 帮助定位 badcase 根因
- 帮助做版本对比

这些不是方案主目标，不应替代 E2E 任务和主评测指标。

### 可观测性工具

| 组件 | 工具 | 部署方式 | 用途 |
|------|------|---------|------|
| LLM 追踪 | Langfuse | Docker Compose 自部署 | Trace / token / prompt / 延迟 |
| AI 质量评测 | Phoenix | pip install | 幻觉检测 / 质量趋势 |
| 指标采集 | OpenTelemetry SDK | pip install | 统一 telemetry |
| 指标存储 | Prometheus | Docker | 时间序列指标 |
| 可视化 | Grafana | Docker | Dashboard |
| 告警 | Prometheus AlertManager | Docker | 阈值告警 |

Langfuse 追踪自动捕获：input/output、Token 消耗、延迟、模型名、session_id、每步 tool_call。

### 记忆系统

```
Layer 1: 会话记忆 (Session Memory)       — 已实现
Layer 2: 用户画像 (User Profile)          — 已实现(基础版)
Layer 3: 语义记忆 (Semantic Memory)       — 待实现，用 SQLite FTS5
Layer 4: 自动摘要 (Auto Summary)          — 待实现
```

语义记忆不引入向量数据库，用 SQLite FTS5 全文搜索 + importance 排序 + 时间衰减。

### 告警规则

- LLM API 错误率 > 10% → critical
- Agent P95 延迟 > 10s → warning
- 幻觉率 > 10% → critical
- 评测通过率 < 85% → warning

### DeepSeek 配置

```env
DEEPSEEK_API_KEY=***
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
```

注意：graph.py 中 DeepSeek 的 base_url 硬编码了 `/v1` 后缀，需改为从 config 读取。

## 改造前后对比

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 核心架构 | V2 Pipeline (5节点串行) | V3 Agent (ReAct循环) |
| LLM角色 | 翻译官(填模板) | 决策中枢(自主推理) |
| 输出方式 | 固定模板报告 | 自然语言建议 |
| 角色处理 | 4套固定模板 | 按问题动态适配 |
| 评测体系 | D1-D8八维(部分虚) | 4+2指标(全实) |
| 评测集 | golden_set.json单文件 | 三层防污染体系 |
| 温度配置 | 全局单一 | 按任务场景5档 |
| PE设计 | 100行固定模板 | 约束边界+工具指引 |
| LLM Provider | 单一百度千帆 | 5家自动切换 |
| 记忆系统 | 无 | 会话+用户画像+语义记忆 |
| 可观测性 | 无 | Langfuse+Phoenix+Prometheus |
| CI/CD | 无 | pre-push+GitHub Actions |
| badcase管理 | 无 | 自动沉淀+回归验证 |
