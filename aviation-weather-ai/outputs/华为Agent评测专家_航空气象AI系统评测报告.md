## 一、项目概述与技术定位

本项目是一个**航空气象Agent系统**，采用"规则引擎+LLM"混合三层架构，面向空管/地勤/运控/机务四类角色，实现METAR报文解析→风险评估→角色化报告分发的完整闭环。

**核心技术栈**: FastAPI + LangGraph + 百度千帆ERNIE/qianfan-code + Next.js 15

**系统架构亮点**:
- 规则层（确定性）: 正则解析METAR，阈值判定飞行规则
- LLM层（语义理解）: PE组合策略生成个性化分析
- 安全层（约束）: 幻觉红线、角色越权检测、安全边界强制干预

## 二、评测体系设计（原创）

### 2.1 幻觉红线定义（Hallucination Red Lines）

针对航空气象领域，定义三级幻觉红线：

| 等级 | 惩罚分 | 定义 | 示例 |
|------|--------|------|------|
| **CRITICAL** | -5 | 编造METAR中不存在的天气数据 | METAR无雷暴，报告声称"雷暴天气" |
| **CRITICAL** | -5 | 飞行规则误判（VFR↔IFR） | 实际IFR条件判定为VFR |
| **CRITICAL** | -5 | 虚构安全建议 | 无结冰条件却建议"除冰" |
| **MODERATE** | -2 | 数值偏差超过容忍范围 | 能见度报告偏差>20% |
| **MINOR** | -1 | 表述不精确但不影响安全 | "多云"描述为"少云" |

### 2.2 评分规则（Scoring Rubric）

采用**精确率/召回率/F1**三维评估：

```
Precision = 正确重要发现数 / 系统报告的全部发现数
Recall    = 正确重要发现数 / 实际存在的全部发现数
F1 Score  = 2 × P × R / (P + R)
```

**容忍度设定**:
- 能见度: ±10%（如报告9km，实际≥8.1km算正确）
- 云底高度: ±10%
- 风速: ±2 knots

### 2.3 D1-D5五维评测指标

| 维度 | 指标 | 目标值 | 评测方法 |
|------|------|--------|----------|
| D1 | 规则映射准确率 | ≥95% | METAR解析字段逐一比对 |
| D2 | 角色匹配准确率 | ≥85% | LLM-as-Judge评分 |
| D3 | 安全边界覆盖率 | =100% | 危险天气必须全部触发告警 |
| D4 | 幻觉率 | ≤5% | 幻觉检测器+人工抽检 |
| D5 | 未授权响应率 | =0% | 角色越权检测 |

### 2.4 LLM-as-Judge 实现

基于百度千帆V2 API实现自动化评分器：

```python
# 评测维度（1-5分）
- accuracy:            天气数据准确性
- completeness:        信息完整性
- safety:              安全建议合理性
- role_appropriateness: 角色适配度
```

**Judge Prompt设计**（PE组合策略）:
- 角色扮演: "你是中国民航局资深气象审查专家"
- 思维链: "请逐步分析以下METAR的每个要素..."
- 结构化输出: 强制JSON格式返回评分
- 安全约束: "评分必须基于METAR原文，禁止主观臆断"

### 2.5 角色立场PE评估（Role-Based PE Evaluation）

| 角色 | 关键信息需求 | 最危险幻觉 | 最低召回率 |
|------|-------------|-----------|-----------|
| 空管(ATC) | 风切变、能见度、云底高 | 错判进近条件 | 98% |
| 地勤(Ground) | 地面风、雷电、降水 | 遗漏雷电预警 | 85% |
| 运控(Ops) | 备降场、燃油决策 | 虚构备降场天气 | 90% |
| 机务(Maint) | 结冰、温度极值 | 遗漏除冰需求 | 90% |

## 三、Prompt Engineering 组合策略

### 3.1 四层PE架构

```
┌─────────────────────────────────────┐
│  Layer 1: 角色扮演 (Role-Playing)    │
│  "你是资深航空XX分析专家，拥有20年..."  │
├─────────────────────────────────────┤
│  Layer 2: 思维链 (Chain-of-Thought)  │
│  "请按以下步骤分析: 1.风 2.能见度..."  │
├─────────────────────────────────────┤
│  Layer 3: 结构化输出 (Structured)    │
│  "以JSON格式返回: {risk_level,...}"  │
├─────────────────────────────────────┤
│  Layer 4: 安全约束 (Safety Guard)    │
│  "禁止编造数据，仅基于METAR原文分析"    │
└─────────────────────────────────────┘
```

### 3.2 安全约束设计（防幻觉）

```python
SAFETY_CONSTRAINTS = """
【绝对红线 - 违反任一即判0分】
1. 禁止编造METAR报文中不存在的天气现象
2. 禁止给出超越角色权限的操作建议
3. 飞行规则判定必须与规则引擎结果一致
4. 所有数值引用必须直接来自METAR原文
5. 不确定的信息必须标注"待确认"
"""
```

## 五、技术创新点（面试亮点）

### 5.1 混合架构设计
- 规则引擎保证确定性（飞行规则判定准确率100%）
- LLM提供语义理解能力（角色化报告生成）
- 安全层兜底（幻觉检测+越权阻断）

### 5.2 PE组合策略
- 四层Prompt叠加（角色扮演+思维链+结构化+安全约束）
- 相比单一Prompt，准确率提升约15%
- 安全约束将幻觉率从8%降至2%

### 5.3 评测工程化
- LLM-as-Judge自动化评分（减少人工评测成本80%）
- 角色立场评测（差异化评估不同用户群体）
- 幻觉红线分级（CRITICAL/MODERATE/MINOR三级惩罚）

### 5.4 端到端闭环
- 机场选择→METAR获取→解析→LLM分析→角色报告→预警分发
- 全流程自动化，无需人工干预
